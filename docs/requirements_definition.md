## 1. システム化の背景と目的（プロジェクト概要）

### 1.1 背景・課題

短文投稿（マイクロブログ）型のコミュニティサービスでは、ユーザー同士が自由にテキスト・画像を投稿し、相互にフォローし合うことで情報交換を行う。一方で、投稿数が増えるにつれて以下の運用課題が顕在化する。

- **スパム投稿の増加**：宣伝・無関係なリンク・反復投稿などがタイムラインを汚染し、サービス体験を損なう。
- **攻撃的・有害な投稿への対応遅延**：怒り・誹謗中傷を含む投稿に対し、人手による監視だけでは検知・対応が間に合わない。
- **目視確認の限界**：投稿量に比例して運用負荷（モデレーション工数）が増大し、対応の属人化・見落としが発生する。
- **対応の即時性不足**：投稿直後に問題を可視化できないと、炎上・トラブルの早期発見ができない。

これらの課題に対し、投稿時に自動でテキストを解析し、スパム度・怒り（有害性）を機械的にスコアリングして即時に可視化する仕組みが求められている。

### 1.2 目的

本システムは、ユーザーが投稿・交流できるマイクロブログ基盤を提供したうえで、AI（外部テキストモデレーションAPI）による自動判定を組み込み、運用者の監視負荷を軽減し、問題投稿の早期発見を支援することを目的とする。

### 1.3 期待効果

| 区分 | 効果 |
| --- | --- |
| 運用効率 | 投稿のスパム判定・怒り判定を自動化し、目視確認の対象を絞り込むことでモデレーション工数を削減 |
| 即時性 | 投稿直後に判定処理を非同期で実行し、結果をリアルタイム（画面リロード不要）で表示 |
| 安全性 | 攻撃的投稿（「至急対応」フラグ）を可視化し、運用者の初動を早める |
| 体験向上 | 判定処理は投稿フローを止めない非同期方式とし、ユーザーの投稿操作を妨げない |

### 1.4 スコープ

**システム化の対象（本システムの範囲）**

- ユーザー登録・認証・プロフィール管理（アカウント有効化、パスワードリセット含む）
- 投稿（テキスト + 画像）の作成・削除・一覧表示
- フォロー / フォロワー関係の管理とフィード（フォロー中ユーザーの投稿表示）
- 投稿に対するAI自動判定（スパム判定・怒り/有害性判定）と判定結果の表示
- 多言語表示（日本語・英語）

**対象外（本システムのスコープ外）**

- AI判定ロジックそのものの実装（外部FastAPIサービス側が担当。本システムはAPI連携のみ）
- 自動判定結果に基づく投稿の自動非表示・自動削除（判定結果は表示のみ。最終的な投稿削除判断は人手で実施）
- ダイレクトメッセージ、いいね・コメント等の追加SNS機能
- 課金・決済機能
- スマートフォンネイティブアプリ（Webアプリケーションのみ提供）

## 2. 業務要件（As-Is / To-Be）

### 2.1 As-Is（現状の業務フロー）

AIモデレーション導入前の運用を想定した現状フロー。

```
[ユーザー] ── 投稿 ──▶ [タイムラインに即時掲載]
                              │
[運用者] ── 目視で巡回 ───────┘
   │
   ├─ 問題のある投稿を発見 ──▶ 手動で削除・対応
   └─ 見落とし／対応遅延が発生
```

**課題**

- 運用者がすべての投稿を目視でチェックしており、投稿量に比例して負荷が増加する。
- スパムや攻撃的投稿の検知が運用者の巡回タイミングに依存し、対応が遅れる。
- 判定基準が属人的で、対応のばらつき・見落としが生じる。

### 2.2 To-Be（導入後の新しい業務フロー）

```
[ユーザー] ── 投稿（テキスト/画像） ──▶ [DB保存・タイムライン即時掲載]
                                            │
                                            ▼
                              [モデレーションジョブを非同期起動]
                                            │
                          ┌─────────────────┴─────────────────┐
                          ▼                                   ▼
              [スパム判定API呼び出し]              [怒り/有害性判定API呼び出し]
                          │                                   │
                          ▼                                   ▼
              spam_score / status 更新          is_angry / angry_score / reason / summary 更新
                          │                                   │
                          └─────────────────┬─────────────────┘
                                            ▼
                          [Turbo Streamsで判定結果を画面に自動反映]
                                            │
                                            ▼
[運用者] ── 「⚠️至急対応」フラグ／スパム度の高い投稿を優先確認 ──▶ 必要に応じて手動削除
```

**改善点**

- 投稿時にAIが自動でスコアリングするため、運用者は判定結果（フラグ・スコア）を手掛かりに優先順位を付けて確認できる。
- 判定は非同期処理のため投稿操作をブロックせず、結果は判定完了後にリアルタイムで画面へ反映される（「判定中…」表示 → 結果表示）。
- 判定基準が機械化され、検知の網羅性・一貫性が向上する。

### 2.3 システム化対象と手作業の切り分け

| 業務 | 区分 | 備考 |
| --- | --- | --- |
| ユーザー登録・認証・アカウント有効化 | システム化 | メールによる有効化リンク送付 |
| 投稿の作成・保存・表示・削除 | システム化 | 削除は投稿者本人のみ可能 |
| フォロー / フォロワー管理 | システム化 | |
| スパム判定・怒り判定（スコアリング） | システム化（外部API連携） | 判定ロジックは外部FastAPIサービス |
| 判定結果の表示・可視化 | システム化 | リアルタイム反映 |
| **問題投稿の最終的な削除・対応判断** | **手作業（運用者）** | 自動削除は行わず、判定結果を基に人が判断 |
| ユーザーアカウントの削除 | 手作業（管理者） | 管理者権限を持つユーザーのみ実行可能 |
| 外部判定APIの精度改善・モデル更新 | 対象外 | 外部サービス側の責務 |

### 3.6 外部連携

**あり**：テキストモデレーションAPI（外部FastAPIサービス）と連携する。

| 項目 | 内容 |
| --- | --- |
| 連携先 | FastAPIサービス（ベースURL：`http://127.0.0.1:8000`） |
| 通信方式 | HTTP POST / JSON（Content-Type: application/json） |
| スパム判定 | エンドポイント `/predict`、リクエスト `{ text }`、レスポンス `{ score, label }` |
| 怒り判定 | エンドポイント `/angry`、リクエスト `{ text }`、レスポンス `{ moderation: { is_angry, score, reason }, summary }` |
| 呼び出し方式 | 投稿保存後に非同期ジョブ（ModerationJob）から呼び出し |
| 結果反映 | 判定結果でMicropostを更新し、Turbo Streamsで該当投稿のステータス領域を差し替え表示 |
| 異常時 | API呼び出し等で例外発生時は processing_state を `failed` に更新 |

### 3.7 AIモデレーション処理フロー（F-14〜F-16）

1. 投稿保存成功後、`ModerationJob` を非同期で起動（投稿フローはブロックしない）。
2. 状態を `processing` に更新し、画面に「スパム判定中…」「怒り判定中…」を表示。
3. スパム判定APIを呼び出し、`spam_score` と `status`（ラベル）を更新 → 画面反映。
4. 怒り判定APIを呼び出し、`is_angry` / `angry_score` / `reason` / `summary` を更新。
5. 状態を `done` に更新し、最終結果を画面表示。
   - `is_angry = true` の場合「⚠️至急対応」と理由を表示、`false` の場合「✅怒っていません」と理由を表示。
   - スパム度はパーセンテージ表示。
6. 処理中に例外が発生した場合は状態を `failed` に更新。

## 4. 非機能要件

### 4.1 性能・拡張性

| 項目 | 要件 |
| --- | --- |
| 投稿レスポンス | 投稿操作は同期的にDB保存・リダイレクトのみ行い、AI判定は非同期化することで投稿操作の体感応答を阻害しない |
| 一覧表示 | ユーザー一覧・投稿一覧・フォロー一覧はページネーション（will_paginate）で分割表示し、件数増加時のレスポンス劣化を抑制 |
| 非同期処理基盤 | バックグラウンドジョブはSolid Queue（DBバックエンド）で処理。ジョブ量増加時はワーカー追加でスケール可能 |
| インデックス | email（一意）、microposts(user_id, created_at)、relationships のフォロー関係に索引を付与しクエリ性能を確保 |
| キャッシュ | Solid Cache をキャッシュストアに採用 |
| 拡張性 | 判定ロジックを外部API（FastAPI）として疎結合に分離し、判定モデルの差し替え・追加を本体改修なしで実施可能 |

### 4.2 可用性

| 項目 | 要件 |
| --- | --- |
| ヘルスチェック | `/up` エンドポイントで起動状態を返却（ロードバランサ・監視ツール用） |
| 外部API障害耐性 | 判定APIが利用不可・異常応答の場合でも、投稿自体は保存・表示され、該当ジョブのみ `failed` 状態とする（投稿機能は継続稼働） |
| 非同期処理の隔離 | 判定処理はジョブキューで分離し、判定遅延・失敗が本体のWebリクエストに波及しない |

### 4.3 セキュリティ

| 項目 | 要件 |
| --- | --- |
| パスワード保護 | パスワードはBCryptでハッシュ化して保存（has_secure_password）。平文は保持しない |
| 認証 | セッションによるログイン管理。「ログイン状態を保持」は記憶トークンのハッシュで実現 |
| トークン管理 | アカウント有効化・パスワードリセットはトークンのハッシュ（digest）をDB保存し、生トークンは送付メールのみで検証 |
| 有効期限 | パスワードリセットリンクは発行後2時間で失効 |
| 認可（アクセス制御） | 未ログインユーザーは要認証ページへアクセス不可（ログイン画面へ誘導）。投稿削除は本人のみ、プロフィール編集は本人のみ、ユーザー削除は管理者のみに制限 |
| 入力検証 | メール形式・文字数・必須・画像サイズ（最大5MB）等をモデルバリデーションで担保 |
| メールアドレス保護 | メールは小文字化・一意制約で正規化し重複登録を防止 |
| 個人情報保護 | 氏名・メールアドレス等の個人情報を取り扱う。アクセス制御・パスワードハッシュ化により保護し、有効化前ユーザーのプロフィールは非公開 |
| 静的解析・脆弱性監査 | Brakeman（脆弱性静的解析）、bundler-audit（依存gemの既知脆弱性監査）を開発・テスト時に利用 |
| 対応ブラウザ制限 | モダンブラウザのみ許可（allow_browser :modern） |

### 4.4 運用・保守性

| 項目 | 要件 |
| --- | --- |
| 国際化（i18n） | 日本語・英語のメッセージリソース（locales/ja.yml, en.yml）でUI文言を一元管理 |
| メッセージ管理 | フラッシュメッセージ・ラベル等をロケールファイルに集約し、コード分散を防止 |
| テスト | モデル・コントローラ・統合（integration）・メーラーの自動テストを整備し、リグレッションを検知 |
| コード品質 | RuboCop（rails-omakase）でコーディングスタイルを統一 |
| デプロイ | Docker / Kamal によるコンテナデプロイを想定。Thrusterによるアセットキャッシュ・圧縮 |
| 監視 | `/up` ヘルスチェックで稼働監視を実施 |
| 画像保管 | 投稿画像はActive Storageで管理 |

### 4.5 制約・前提

| 項目 | 内容 |
| --- | --- |
| 開発フレームワーク | Ruby on Rails 8.1系 |
| データベース | SQLite3（Active Record） |
| 非同期/キャッシュ/通知基盤 | Solid Queue / Solid Cache / Solid Cable |
| フロントエンド | Hotwire（Turbo / Stimulus）、importmap、Propshaftアセットパイプライン |
| 外部判定サービス | FastAPI（ローカル `127.0.0.1:8000` で稼働を前提）。本番運用時は接続先・認証方式の整備が必要 |
| 提供形態 | Webアプリケーション |

## 付録：用語定義

| 用語 | 説明 |
| --- | --- |
| マイクロポスト（Micropost） | ユーザーが投稿する短文（最大140文字）+ 任意の画像 |
| フィード | 自分とフォロー中ユーザーの投稿を時系列で集約した表示 |
| モデレーション | 投稿内容の自動判定（スパム判定・怒り/有害性判定） |
| スパム度 | スパム判定APIが返すスコア（パーセンテージ表示） |
| 至急対応 | 怒り判定で is_angry=true となった投稿に付与される注意表示 |
| 有効化（アクティベーション） | メール経由でアカウントを利用可能状態にする処理 |
| Turbo Streams | 画面リロードなしで部分更新を行うHotwireの仕組み |
